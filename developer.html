Deepward Developer Codex

The complete code-level guide to extending, balancing, testing, and maintaining Deepward. Deepward is a dependency-free browser RPG built with ordered global JavaScript modules, HTML-string rendering, local storage, and CSS.

IDs are permanent API. A renamed trait, node, class, enemy, recipe, slot, or stat can break saves, prerequisites, crafting, and balance metrics.

1. Start here

Open index.html directly or serve the repository through any static HTTP server. There is no build step, dependency install, framework, database, or backend.

Repository

  • index.html: document shell and ordered script loader.
  • ui/deepward-style.css: complete visual system, layout, responsive behavior, and animation.
  • core/*.js: game definitions, generators, state, systems, UI, saving, and telemetry.
  • README.md: player-facing overview.
  • developer.html: this technical guide.

Architecture rules

  • Definitions are plain objects and arrays. Systems interpret those definitions.
  • STATE is the single live model.
  • Gameplay mutations belong in Engine or another owning system.
  • UI callbacks translate browser input, call a system, and render.
  • Combat-only data belongs under state.combat; run-only data belongs under state.dungeon.
  • Call render() after user-visible state changes.

Scripts expose globals instead of ES modules, so index.html loads them sequentially. A module must appear after everything it reads during evaluation.

→ metrics → combat_engine → save_system → renderer → event_wiring

4. Stats and balance

stats.js defines CORE_STATS, ELEMENTS, generated elemental stats, and lookup tables. Every item stat must exist in STAT_BY_ID.

StatsPurpose
ATK / MATKPhysical and magical attack inputs.
DEF / MDEFPhysical and magical defense inputs.
SPDInitiative and fleeing.
HIT EFF / HIT RESHit chance, avoidance pressure, and critical contribution.
HP / MPFlat equipment resource-cap additions.
Element DmgPercentage bonus for its damage type.

Engine.computeDerived combines base formulas, level growth, class modifiers, item stats, skill passives, conversions, and dungeon modifiers. Engine.effectiveStat layers temporary combat percentages on the sheet. Engine.statModifiers supplies visible temporary-source tooltips.

Current scaling contract

  • Item rolls use level ^ BALANCE.itemLevelExponent, then apply stat and rarity multipliers.
  • BALANCE.damageMitigation(defense, attackPower) returns proportional reduction with a hard cap; defense cannot reduce ordinary damage to a permanent one-damage floor.
  • Hit chance is bounded by BALANCE.hitChance, while HIT EFF contributes to critical chance through a square-root curve and final critical chance is capped.
  • BALANCE.difficultyMultiplier combines the selected base multiplier with level-ramped endgamePressure.
  • Pack frequency ramps through monsterPackChance, and boss HP gains a separate late-game bossDurability curve.
  • Town material bundles use BALANCE.materialBundleCost(level); price targets the expected number of random bundles needed to satisfy one authored Mythic recipe.

Put adjustable numbers in BALANCE. Test early, middle, and late levels whenever changing a curve. Balance against exported samples, not one unusually lucky run.

5. Gear and loot

{ uid, name, slot, slotLabel, tier, ilvl, stats,
  uniqueTraits: [], mythicTrait: null, element, crafted? }

TIERS defines stat count, trait count, rarity weight, stat multiplier, roll range, and trait multiplier. Traits begin at Ultra Rare. SLOTS defines the ten equip positions and their favored stat rolls.

Item generation

  1. rollTier applies base and difficulty-adjusted rarity weights.
  2. pickSlot selects an equipment position.
  3. rollStatsForSlot makes distinct weighted stat picks.
  4. lootAffinities favors class stats and learned route damage types.
  5. rollTraits filters the pool by exact tier and allowed slot.
  6. nameItem combines tier prefix, slot noun, and high-tier suffix.

Adding a slot requires changes to SLOTS, initial equipment state, save restoration, naming data, exclusive trait pools, and equipment UI assumptions.

6. Traits and effects

{ id, name, type, base, perLvl, tier?, allowedSlots?, stackGroup?,
  source?, target?, element?, desc: value => 'description' }

Random drops use GEAR_TRAITS; random Mythics use MYTHIC_TRAITS; Soulforge gear uses CRAFTED_MYTHIC_TRAITS. Legacy UNIQUE_TRAITS also supplies fixed crafted properties.

A trait is functional only if its type is consumed by derived-stat calculation, rewards, a dedicated engine check, or EFFECT_HANDLERS.

Combat effect context

{ direction, damage, attacker, target, targetName, source,
  player, derived, maxHp, maxMp, combat, combo, isCrit,
  element, kind, notes }

Handlers receive (ctx, value, trait); use the full trait object for fields such as cadence, cap, threshold, drawback, or cooldown reduction. Generated effects sharing a stackGroup are combined by Engine.consolidateItemTraits. The first copy applies at full value, later copies contribute diminishing value, and type-specific caps protect chance and multiplier effects. Crafted traits without a group remain independent. Legacy generated trait IDs are recognized so imported older saves receive the same stacking rules.

  • Always check direction before applying offensive or defensive logic.
  • Cap permanent battle stacks and define reset/consumption rules.
  • Use combat fields for battle-only counters.
  • Account for bosses, multi-target fights, misses, overkill, and death.
  • Record secondary damage and healing in metrics.
  • Make the tooltip match the actual order, cap, scope, and duration.

7. Soulforge crafting

CRAFTING_MATERIALS is the material registry. MYTHIC_RECIPES describes discovery IDs, costs, slots, elements, fixed stats, Unique properties, and one exclusive Mythic effect.

{ id, name, slot, element, mythicTrait,
  uniqueTraits: ['trait_id'], requirements: { material_id: count },
  stats: { atk: 29, physicalDmg: 14 } }

Crafting.createItem scales fixed stats by character level and materializes trait descriptions. Crafted Mythic effects never enter random pools.

New-recipe checklist

  1. Add or reuse material IDs and ensure they can drop.
  2. Add a recipe with a stable unique ID.
  3. Add its discovery route through boss/event rewards.
  4. Add the exclusive trait definition.
  5. Add its handler or inline engine consumer.
  6. Test crafting costs, scaling, save/export, equip restrictions, and every combat trigger.

8. Classes, subclasses, and skill webs

Each class owns one 129-node web: an automatically awakened class core, four readable 30-node subclass regions, four two-region hybrid junctions, and four outer keystones. The original 120 stable route-node IDs remain valid for saves and metrics. buildSkillWeb places those nodes on polar rings, connects every route to the core, installs alternate hybrid exits, and creates the outer keystone gates.

A creates active nodes and P creates passive nodes. refineRouteNodes adapts the shared route skeleton into Deepward combat mechanics such as momentum, channeling, defense breaches, affliction contagion, status collapse, mana batteries, and kill-resource loops. makeClass then assigns stable IDs, prerequisites, costs, cooldowns, roles, region metadata, and coordinates.

A('Skill', 14, 'nuke', 'Deal 150% fire damage.', {
  power: 1.5, magic: true, forcedElement: 'fire', guaranteedStatus: true
})
P('Passive', { type:'elementStatusPower', element:'fire', value:20 },
  'Fire secondary effects are 20% stronger.')

Web prerequisites

// Linear connection: one parent
{ requires: 'mage_pyre_4' }

// Hybrid junction: both regions must reach the gate
{ requiresAll: ['mage_pyre_x5', 'mage_frost_x5'] }

// Alternate exit: either the native route or hybrid can reach it
{ requiresAny: ['mage_pyre_x12', 'mage_web_hybrid_1'] }

Engine.skillRequirements normalizes all three forms. requiresAll has priority over requiresAny, which has priority over legacy requires. Every referenced ID must exist in the same class web. Never rename an existing ID; add a migration only when removal is unavoidable.

Node roles in game context

RolePurpose
coreFree class anchor automatically present on new and restored characters.
root / spineReadable early route identity and foundational actives.
minorConnective stats or focused support; keeps path decisions legible.
notable / capstoneStronger mechanics and route-defining actions deeper in a region.
hybridRequires investment in two adjacent regions and opens an alternate path.
keystoneExpensive outer rule change with a major payoff and, where appropriate, a drawback.

An active action needs a matching SKILL_ACTION_HANDLERS entry. A passive effect needs derived-stat, dedicated engine, or EFFECT_HANDLERS support. Powerful mechanics belong deeper in the web and cost more than connective nodes. Test each route alone, adjacent hybrid paths, multi-region stacking, status-free targets, bosses, resource extremes, and imported saves.

10. Enemies and bosses

MONSTER_TEMPLATES and BOSS_TEMPLATES define ratios, moves, elements, and flavor. Move helpers create strikes, buffs, heals, and debuffs. ENEMY_IDENTITIES adds an authored tactical role, stat modifiers, charge bias, and utility bias.

{ id, name, icon, element, flavor, atk, def, matk, mdef, spd,
  moves: [Strike('Move', 1.2, { dotPct: 5 })] }
Identity('Tactical role', 'Player-facing explanation',
  { atk:1.15, hp:.9, spd:1.1 }, chargeBias, utilityBias)

monsterFromTemplate combines level scaling, the effective difficulty multiplier, early-game mercy, identity modifiers, boss durability, and per-spawn variance. generateBattleGroup uses the level-ramped pack chance, so early encounters favor readable single enemies while later floors apply more group pressure. Every new enemy needs an identity. Test its minimum and maximum rolls, solo and pack behavior, charge/utility cadence, status pressure, elemental matchup, and boss phase.

11. Combat lifecycle

  1. startCombat refreshes stats and creates encounter state.
  2. The player locks an action.
  3. resolveRound sorts living actors by SPD plus bounded jitter.
  4. Actors resolve sequentially with visible delays.
  5. resolveAttack handles hit, crit, defense, elements, traits, HP, ailments, and telemetry.
  6. endOfRound ticks regeneration, DoTs, ailments, cooldowns, statuses, retargeting, revival, and victory.
  7. endCombat records the result and advances room flow.

Damage order

attack power → proportional defense mitigation → element bonus/matchup → critical
→ outgoing traits → HP damage → ailments and secondary hits

incoming roll → defensive traits → Defend mitigation → Radiant Ward → HP

Ordering is part of balance. Document intentional changes. Gear changes and skill learning are forbidden in combat. Controls remain disabled while a round is resolving.

12. Dungeons and events

generateDungeon chooses theme, difficulty, room count, weighted rooms, pacing corrections, two Depth Laws, and a boss. enforceCombatPacing limits non-combat streaks and enforces a minimum fight share. A three-option boon draft occurs after every second cleared non-boss room, and selected boons persist until the run ends.

Boons and upgrades

DUNGEON_BOONS defines each rank-I family. BOON_POWER_BY_ID assigns its intrinsic lesser, greater, or mythic power rarity; BOON_POWER_TIERS supplies base draw weights. upgradedDungeonBoon(base,tier) materializes ranks I–III and uses BOON_UPGRADE_TEXT for rank-specific descriptions. Power rarity and upgrade rank are separate: rarity controls how often a family is offered, while rank describes the owned chain level.

Engine.offerBoonChoices uses weighted sampling without replacement. Greater and Mythic weights increase with difficulty.boonRarityBonus, while Lesser weight decreases. Eligible rank-II and rank-III entries enter the same pool only when the preceding rank is owned; their weights are additionally reduced, so an upgrade is possible but never guaranteed. Claiming an upgrade replaces the prior rank instead of stacking it.

EVENT_HANDLERS covers battles, chests, merchants, gambling, traps, shrines, mysteries, bosses, ambushes, archives, wells, collapses, healers, altars, bonfires, black markets, puzzles, foraging, guarded caches, deep pools, and camps.

Add a boon

  1. Add a stable-ID rank-I definition to DUNGEON_BOONS.
  2. Assign the ID in BOON_POWER_BY_ID based on total run impact, not numerical size alone.
  3. Add both higher-rank descriptions to BOON_UPGRADE_TEXT.
  4. Implement exceptional scaling in upgradedDungeonBoon; generic numeric effects may use its default scaling.
  5. Register any new effect type in the appropriate combat trait hook or add the temporary skill action handler.
  6. Test all three ranks, replacement behavior, battle resets, run cleanup, and draft frequency on every difficulty.

Add an event

  1. Add its room weight in the generator.
  2. Add EVENT_HANDLERS[type](state).
  3. Log the event and set choices or begin combat.
  4. Ensure all branches eventually finish the room or enter a terminal flow.
  5. Choose the correct duration for rewards and penalties.
  6. Test insufficient resources, repeated events, save boundaries, and defeat paths.

13. Rendering and UI

renderer.js builds HTML strings. render() chooses the scene, updates #app, applies theme classes, saves, animates bars, and upgrades title attributes into custom tooltips.

The skill-web overlay renders region backdrops, every normalized prerequisite edge, node-role seals, a persistent inspector, and region-focus navigation. Selection and viewport scroll live in state.ui; rerenders restore the previous map position. Locked nodes remain selectable so their mechanic and missing prerequisites can be inspected.

Global callbacks live in event_wiring.js because inline event attributes need global functions. Keep callbacks thin and escape player-controlled or imported strings with U.escapeHtml.

Overlay contract

  • Open/closed state lives in state.ui.
  • The backdrop closes the overlay.
  • Mutation restrictions are enforced in the engine as well as UI.
  • Combat controls disable during resolution.
  • New animation respects reduceMotion.
  • Layouts are tested on narrow screens.

Reuse CSS variables, tier colors, panels, buttons, typography, tooltip behavior, and animation language in deepward-style.css.

14. Saving and migration

SaveSystem stores deepward-save-v1 in localStorage and exports JSON. Snapshots exclude live dungeon and combat state so loading resumes safely in town.

validate rejects unrelated or future-version saves. restore sanitizes strings, validates item slots and stats, filters unlocked node IDs against the selected class, inserts the free class-core ID when absent, reconstructs equipment, defaults fields, clears transient flags, recomputes derived stats, and refills resources. Keeping all pre-web route IDs stable makes earlier saves compatible without refunding or remapping their nodes.

Schema changes

  1. Increment the version for incompatible changes.
  2. Migrate older payloads before restoration.
  3. Default every new optional field.
  4. Never serialize callbacks, timers, DOM nodes, functions, or circular combat references.
  5. Test local load, export/import, invalid JSON, missing fields, and an older fixture.

15. Balance metrics

Telemetry is local-only under deepward-balance-metrics-v1. Nothing leaves the browser unless the player exports it. Metrics cover totals, actions, elemental results, loot tiers, difficulty outcomes, combat rounds, and performance for classes, subclasses, nodes, skills, items, traits, and mobs.

Web-node records include unlock count, primary region, all attributed regions, node role, kind, and effect/action type. Hybrid nodes contribute to both adjacent subclass samples; the class core does not create a fake subclass record. Record at the owning boundary: battle start/end, dungeon outcome, node unlock, mob action, skill use, rewards, loot decisions, damage, and healing. Use stable IDs for keys.

Analysis guidance

  • Compare win rate and average rounds by level and difficulty.
  • Use mob damage caused/received and player deaths to identify outliers.
  • Compare skill use, damage, healing, kills, and MP efficiency.
  • Read trait or node win rate alongside pick rate and sample size.
  • Compare keystone and hybrid performance against their point gates, not early minor nodes.
  • Do not balance around tiny samples or one optimized build.

17. Safe change workflow

  1. Find the definition and every interpreter that consumes it.
  2. Choose a stable ID and the correct lifetime.
  3. Add data and engine behavior together.
  4. Add UI feedback: description, tooltip, tag, log, disabled state, or animation.
  5. Add telemetry at the actual gameplay boundary.
  6. Evaluate save and migration impact.
  7. Run syntax, whitespace, registry, prerequisite, and collision checks.
  8. Test misses, criticals, overkill, death, bosses, packs, stacking, and resource extremes.
  9. Update this guide when a developer contract changes.
Never add a trait type, skill action, monster move, room type, stat, or save field as data only. Every new data contract needs an interpreter and usually UI, metrics, and persistence work.

22. Crafting, saving, and metrics APIs

Crafting

SignatureContract
canCraft(state,recipe)Returns true only if recipe is known and every material cost is met.
createItem(recipe,level)Pure item construction; does not spend resources or grant the item.
craft(state,recipeId)Returns boolean; spends materials, creates item, grants inventory, logs result.

SaveSystem

SignatureContract
snapshot(state)Returns portable town-safe payload; throws without active player.
validate(data)Returns input or throws an Error for foreign, malformed, or future data.
save(state)Returns boolean; catches localStorage errors.
hasSave()Boolean local-save presence check.
readLocal()Returns validated parsed payload or throws.
restore(data)Sanitizes and installs global STATE, saves it, and renders.
loadLocal()UI-safe wrapper that alerts on error.
export(state)Downloads formatted JSON.
importFile(file)Asynchronously reads, parses, validates, and restores a browser File.

Metrics

CallWhen to use
addTotal(name,value=1)Global additive counters.
count(group,key,value=1)Dynamic grouped counters such as ailments or actions.
battleStarted(state,monsters)Exactly once after combat model creation; returns build snapshot.
battleEnded(state,result,combat)Exactly once before combat is discarded.
combatFlow(combat,direction,amount,mob)Every actual dealt/taken damage event.
healing(combat,amount)Actual healing, not attempted healing.
mobAction(mob,type)Type is turn, charge, or utility.
skillPerformance(skill,data)Data supports damage, healing, kills, and mana.
reward(gold,xp)When currency/XP is actually granted.
loot(item,decision)Decision is offered, taken, or left.
